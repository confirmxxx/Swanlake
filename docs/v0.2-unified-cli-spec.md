# Swanlake v0.2 — Unified CLI Spec

> Status: **shipped in v0.2.0**, additive surface in v0.3.0. Preserved as the design source of truth for the v0.2 unified CLI slice; the v0.3 deploy/sweep additions are documented separately in [`v0.3-beacon-deploy-spec.md`](v0.3-beacon-deploy-spec.md). Inline references to the "v0.2 release" should be read in that historical context — the CLI surface here remains current except where noted.

## Headline

Through v0.1 the operator interacted with seven scattered entry points across three filesystems. v0.2 collapses those into one CLI — `swanlake` — that is harness-agnostic by construction. **Two adapters ship in v0.2**: Claude Code (in-use by the operator) and CMA (validated against synthetic CMA-shape fixtures; first live-project install is operator follow-up work). SDK adapter remains a stub for v0.3 (no current SDK adopters).

The killer feature is `swanlake status`: a single command that aggregates seven independent health dimensions and exits non-zero on any drift. The real failure that motivated this work — the reconciler reporting `overall: missing` only because the operator manually asked — becomes visible at every shell prompt.

## The seven entry points being consolidated

```
python3 -m reconciler.cli --status / --sync / --init
python3 ~/projects/Swanlake/tools/status-segment.py
python3 ~/projects/Swanlake/tools/loop-closure-metric.py
python3 ~/projects/DEFENSE-BEACON/make-canaries.py
python3 ~/projects/DEFENSE-BEACON/verify-beacons.py
~/.claude/hooks/{canary-match,content-safety-check,bash-firewall,exfil-monitor}.sh
/tmp/swanlake-{ab,pyrit-garak}-bench-2026-04-26/run.sh
```

## CLI surface (locked)

```
swanlake status        composite posture across 7 dimensions; exit 0/1/2
swanlake sync          reconcile canon to managed surfaces (confirmation gated)
swanlake verify        check which surfaces still hold intact beacons
swanlake rotate        rotate canary tokens across the registry (DESTRUCTIVE)
swanlake bench         --quick (1 min smoke; v0.2). --full (PyRIT+Garak, ~1 h) is a stub in v0.2 and exits 3 with a manual-fallback hint; ships v0.3.
swanlake doctor        per-primitive health check with fix suggestions
swanlake init          first-run bootstrap; idempotent
swanlake adapt cc      install hooks + status-line + slash command
swanlake adapt cma     install Beacon Part A + zones + reflex-purity check into a CMA project
swanlake adapt sdk     stub — NotImplementedError, exit 3
```

Top-level flags: `--version`, `--state-root PATH` (override `~/.swanlake/`), `--quiet`, `--json` (per-subcommand).

## Architecture decisions (with rationale)

### A1. Stdlib argparse, not click/typer/rich
The repo's design principle is stdlib-first (CONTRIBUTING.md). `reconciler/cli.py` is already argparse and the pattern works. Two-level subparsers handle `swanlake adapt cc` cleanly. No new runtime dependencies.

### A2. Sibling package, not absorption
`reconciler/`, `tools/`, `defense-beacon/` stay where they are. The new `swanlake/` package imports from them. Reasoning: the existing systemd timer (`~/.config/systemd/user/swanlake-watchdog.timer`) calls `python3 -m reconciler.cli --sync`; absorbing reconciler into a deeper namespace would require timer-unit rewrites in v0.2, which is scope creep. v0.3 may consolidate after the timer migration is its own task.

### A3. `~/.swanlake/` as the unified state root
Already exists for the canary registry. Add three new files. Never touch existing files in this dir (specifically `canary-hits/` and `canary-strings.txt` per A11 SPEC).

```
~/.swanlake/
├── config.toml                   # NEW — operator config (supersedes ~/.config/swanlake-reconciler/config.toml)
├── canary-strings.txt            # EXISTING — untouched
├── canary-hits/                  # EXISTING — untouched
├── audit.jsonl                   # NEW — every CLI invocation appends one line
├── coverage.json                 # NEW — inventory of inventories (replaces stale deployment-map.json)
└── last-bench                    # NEW — ISO-UTC timestamp of last successful bench run
```

Config loader reads `~/.swanlake/config.toml` first, falls back to `~/.config/swanlake-reconciler/config.toml` with a one-line stderr deprecation hint.

### A4. Tool module renames (Python-importable)
`tools/status-segment.py` → also installed as `tools/status_segment.py` (importable). Original kept as a shim because `~/.claude/hooks/status-line.sh` invokes it by absolute path. Same treatment for `tools/loop-closure-metric.py`.

### A5. Subprocess wrappers for beacon scripts
`swanlake verify` and `swanlake rotate` call `make-canaries.py` and `verify-beacons.py` via `subprocess.run([sys.executable, str(script_path), ...])`. They do not import them. This keeps the beacon scripts standalone (their `.canary-state.json` lives next to `out/`) and avoids `sys.path` contamination.

### A6. `pyproject.toml` at top level
Console-script entry point: `[project.scripts] swanlake = "swanlake.__main__:main"`. Lists both `swanlake` and `reconciler` as packages so existing `python3 -m reconciler.cli` continues to work post-install. Optional `[bench]` extra for PyRIT+Garak (~500 MB combined; not pulled in by default install).

### A7. Safety-gate model
| Subcommand | Default | Bypass |
|---|---|---|
| `status`, `verify`, `doctor`, `bench`, `adapt cc --verify` | no prompt | n/a |
| `sync` | `[y/N]` showing what will be touched | `--yes` flag OR `SWANLAKE_NONINTERACTIVE=1` |
| `rotate` | `[y/N]` showing surfaces + warning that old tokens stop detecting immediately | same |
| `adapt cc --uninstall` | `[y/N]` showing what will be removed | same |

When NONINTERACTIVE env is active, audit row records `noninteractive: true`.

### A8. The harness-adapter contract
Abstract `Adapter` class with `install / uninstall / verify / list_surfaces`. **Two adapters ship in v0.2**: CC (production-validated against the operator's own setup) and CMA (fixture-validated, awaiting first live-project install). Together they exercise the abstraction enough to catch the worst contract leaks; a third independent adopter remains the real validator. SDK adapter inherits the contract but raises `NotImplementedError` until a real SDK adopter appears.

**CMA adapter scope**:
- Reads a CMA project layout (per-CMA agent definitions — typically a `cmas/` or `agents/` dir with one markdown/yaml file per CMA; configurable glob)
- Installs Beacon Part A operating rules into each CMA's system-prompt source (preserving Part B canary attribution if already present, or generating fresh per-CMA canaries on first install)
- Reads the project's `zones.yaml` (or generates a default classifying each CMA as INTERNAL by default) and applies the zone's tool-allowlist semantics to each CMA's tool config
- Runs `test_reflex_purity.py` style AST check against the project's hot-path code (looking for LLM calls inside files matching `**/reflex*.py`, `**/hot_path*.py`, or operator-configurable globs). The check reports violations to stderr and writes them to the per-project manifest, but does not modify or block the project — `swanlake adapt cma` exits 0 even with violations present. Operator decides whether to wire the reported violations into CI as a hard gate.
- Registers each CMA as a surface in `~/.swanlake/coverage.json` with `type=cma, project=<name>, cma_id=<id>`
- Manifest at `~/.swanlake/cma-adapter-manifest-<project>.json` for idempotent uninstall

### A9. Composite `swanlake status` — 7 dimensions

| Dimension | Source | Severity mapping |
|---|---|---|
| reconciler | `reconciler.status.compute_report()` | fresh=0, drift=1, missing=2 |
| canary | `tools/status_segment.count_today(CANARY_DIR, _canary_hit)` | hits>0 → 2 |
| inject | `tools/status_segment.count_today(CONTENT_DIR, _content_safety_hit)` | hits>0 → 2 |
| exfil | `tools/status_segment.count_today(EXFIL_DIR, _exfil_hit)` | hits>0 → 2 |
| closure | `tools/loop_closure_metric.aggregate_window(today, 7)` | ratio<0.30 → 1 |
| coverage | `~/.swanlake/coverage.json` mtime + content | >7d stale → 1, missing → 2 |
| bench | `~/.swanlake/last-bench` mtime | >30d → 1, missing → informational |

Worst severity wins. Each dimension function returns a typed dict and never raises — failures degrade to `{"status": "unknown", "error": "<msg>"}` and contribute severity 1 (missing tool ≠ attack signal).

Sample human output:
```
swanlake status — 2026-04-26T14:23Z

dimension       status     detail
--------------  ---------  ----------------------------------------
reconciler      drift      notion: missing, claude_md: 26h, vault: fresh
canary          clean      0 hits / 4 fires (24h)
inject          clean      826 hits / 14081 fires (24h, bench day)
exfil           clean      0 hits / 0 fires (24h)
closure         ok         0.94 ratio (7d window)
coverage        stale      8d since last verify (8 of 25 surfaces tracked)
bench           ok         3d since last quick run

overall: DRIFT  [exit 1]
```

`--json` returns the same data structured for `jq`.

### A10. Telemetry contract — `~/.swanlake/audit.jsonl`
One JSON object per line, append-only, atomic write. Schema:
```json
{
  "ts": "2026-04-26T14:23:01.412+00:00",
  "cmd": "status",
  "subcmd": null,
  "args": ["--json"],
  "exit_code": 1,
  "duration_ms": 142,
  "swanlake_version": "0.2.0",
  "python_version": "3.12.3",
  "pid": 12345,
  "tty": true,
  "noninteractive": false,
  "error": null
}
```

Argv elements matching canary shape regexes (per `defense-beacon/reference/canary-match.sh`) get redacted as `REDACTED(type=canary, pos=N)`. Rotation: when `audit.jsonl > 10 MB`, rename to `audit.jsonl.1` (overwriting existing `.1`) before next append. Two files maximum, no cron.

### A11. Migration semantics — no flag day
v0.1 entry points keep working through v0.2 with stderr-only deprecation hints (so cron consumers of stdout are unaffected). Removed in v0.3. `SWANLAKE_NO_DEPRECATION=1` suppresses the hints during transition.

## What this explicitly does not do (NON-GOALS reaffirmed)

- Does not ship a bash firewall, content-safety scanner, or prompt-injection classifier. Those are operator-local CC hooks; `swanlake adapt cc` installs them as templates from `swanlake/adapters/templates/cc/`, but the CLI itself is not a guardrail engine.
- Does not auto-deploy beacons to remote surfaces. `rotate` updates the registry and emits a deployment checklist; the operator pastes.
- Does not own the `~/projects/DEFENSE-BEACON/` repo's structure. The CLI delegates to the scripts there via subprocess.
- Does not run as a daemon. No `swanlake watch` in v0.2 — systemd timer + status-line glyph already cover the "did something break" signal.
- Does not ship a functional SDK adapter (v0.3 work, gated on a real SDK adopter). The CMA adapter ships against synthetic fixtures only; first live-project install is operator follow-up work.
