# Swanlake v0.4 — enforcement layer spec

> Status: design-locked, pre-implementation. Three-layer model (scan / nudge / init) with the auto-install layer (L4) explicitly out of scope. Same architectural rule v0.3 used to bar auto-deploy on cloud surfaces, applied to local project trees.

## Headline

Through v0.3 the operator can sweep, deploy, and verify beacons across surfaces they already know about. Surface discovery still relies on the operator remembering to run `swanlake beacon sweep` after spinning up a new project. There is no passive feedback loop that points out "this project has a `CLAUDE.md` but no beacon attribution and no opt-out marker — you probably want one of those".

v0.4 adds that feedback loop, with the same discipline that bars auto-deploy on remote surfaces in v0.3 — **nothing auto-installs**. The layer that would silently scaffold a beacon block on detect (call it L4) is the obvious next step from L3 (`swanlake init project`), and it is the layer this spec explicitly refuses to ship.

The L4 anti-pattern is the local-tree analog of v0.3's "no auto-deploy to cloud surfaces" rule. Both rest on the same property: the value of attribution comes from the human-verified deployment step. An attacker who compromises an auto-installer can silently scaffold attacker-known beacon content into every project the operator opens; subsequent canary hits flow through values the attacker chose. The defense's whole attribution property collapses to whatever guards the auto-install trigger. The cheapest defense is to never have one.

## Three layers

| Layer | Scope | Mutative? | Trigger |
|---|---|---|---|
| **L1: `swanlake scan`** | Walk `~/projects/*/`, emit per-project audit: has `CLAUDE.md`? has Beacon attribution? has `.swanlake-no-beacon` opt-out? CMA-shaped (i.e. has a `cmas/` dir)? | NO — read-only audit | Operator invocation |
| **L2: SessionStart hook** (`swanlake-session-nudge.sh`) | At every Claude Code session start, detect the project the session is rooted in. If `CLAUDE.md` exists, no beacon attribution, no opt-out marker -> print one-line stderr nudge. Never installs anything. | NO — advisory only | Claude Code SessionStart event |
| **L3: `swanlake init project --type {cc,cma}`** | Scaffold a fresh project with Swanlake pre-installed. Creates `CLAUDE.md` template, `.swanlake-no-beacon` example, optional `cmas/` dir for CMA type. | YES — explicit operator invocation against a target dir | Operator invocation |
| ~~L4: auto-install on detect~~ | ~~On SessionStart, if conditions match, scaffold the beacon block into `CLAUDE.md`~~ | **EXPLICITLY OUT OF SCOPE — see §10 N1** | n/a |

The hook in L2 is not wired by Swanlake unconditionally. It ships as a template script that the operator opts into via `swanlake adapt cc --enable-session-nudge` (a tiny addition to the existing CC adapter — see §6 E5). Without that opt-in, the script sits unused in the templates tree.

## Hard architectural rule

NOTHING in v0.4 auto-installs without explicit per-action confirmation. Per-action means: the operator typed the command that triggered the write, in the active session, against a named target. SessionStart-event triggers are advisory only — the hook prints one stderr line and exits 0. There is no `swanlake adapt cc --auto-scaffold` flag and no L4 module.

Same machine-enforcement story as v0.3's `local_deploy.py` / `remote_deploy.py` split: the package contains an `init/project.py` module (operator-invoked) and a `nudge/` module (read-only). There is no `nudge/auto_scaffold.py`. Adding one requires touching the dispatch table, the §10 non-goals list, and explicitly removing the L4 anti-pattern from this spec. The diff is unmissable in code review.

## CLI surface (locked)

```
swanlake scan                                 walk ~/projects/*; report per-project status; read-only
swanlake init project [DIR] --type {cc,cma}   scaffold a fresh project skeleton in DIR
swanlake adapt cc --enable-session-nudge      drop the SessionStart nudge hook into ~/.claude/hooks/
swanlake adapt cc --disable-session-nudge     remove the nudge hook (manifest-aware)
```

All inherit the v0.2 common parent: `--state-root PATH`, `--quiet`, `--json`. Two-level subparsers mirror the existing `swanlake adapt cc` and `swanlake beacon <op>` shape.

| Subcommand | Positional | Notable flags | JSON shape (top-level keys) | Exit codes |
|---|---|---|---|---|
| `scan` | — | `--projects-root PATH` (default `~/projects`); `--include-cma` (default off); `--filter {all,actionable,clean}` (default `all`) | `projects: [{path, has_claude_md, has_beacon, has_optout, is_cma_shaped, recommended_action}]`, `summary: {n_total, n_actionable, n_clean, n_optout, n_cma}` | 0 always (read-only) |
| `init project` | `DIR` (default `.`) | `--type {cc,cma}` (required); `--force` (overwrite non-empty target); `--name NAME` (project name, default basename of DIR) | `created: [path...]`, `skipped: [{path, reason}]`, `target: <abs-path>`, `type: cc\|cma` | 0 created / 2 USAGE (target non-empty without `--force`) |
| `adapt cc --enable-session-nudge` | — | `--cc-dir PATH`; `--dry-run` | inherits the existing CC adapter shape; manifest gains `session_nudge: <hook-path>` | 0 enabled / 2 USAGE |

The `scan` verb is intentionally **at top level**, not under `beacon`. Reason: scan is broader than beacon — it reports CMA-shape and opt-out state, not just beacon attribution. Putting it under `beacon` would mis-signal scope. Compare: `swanlake beacon sweep` is "find unbeaconed surfaces on the v0.3 surface-type matrix"; `swanlake scan` is "audit every project under `~/projects/`".

## SessionStart hook contract (L2)

The hook is a stdlib-only bash script, drop-in to `~/.claude/hooks/swanlake-session-nudge.sh`, wired to the SessionStart event in `~/.claude/settings.json` by the CC adapter when `--enable-session-nudge` is passed. It MUST NEVER block — only nudge.

### Input

The Claude Code SessionStart event passes a JSON payload on stdin. Per the v0.3-era settings.json contract the adapter already targets, the relevant field is:

```json
{
  "session_id": "<uuid>",
  "transcript_path": "<abs-path>",
  "cwd": "<abs-path>"
}
```

Per Claude Code's documented `${CLAUDE_PROJECT_DIR}` env var (`~/.claude` settings docs), the project root is also exposed via the env. The hook prefers `${CLAUDE_PROJECT_DIR}` (set by the harness), falls back to walking `cwd` upward looking for a `.git/HEAD` to determine repo root.

### Logic

```
1. Resolve project_root:
     - If $CLAUDE_PROJECT_DIR is set and is a dir, use it.
     - Else read cwd from stdin JSON.
     - Walk upward looking for .git/HEAD; if found, use the parent dir.
     - Else use cwd as-is.
2. If project_root is not under $HOME/projects/ (configurable via
   $SWANLAKE_NUDGE_SCOPE), exit 0 silently. Out of scope.
3. If project_root/.swanlake-no-beacon exists at any ancestor up to
   $HOME/projects/, exit 0 silently. Operator opted out.
4. If project_root/CLAUDE.md does NOT exist, exit 0 silently. The hook
   only nudges projects that already use Claude Code agent context.
5. Grep CLAUDE.md for the beacon attribution sentinel
   (`<!-- DEFENSE BEACON v` literal). If present, exit 0 silently.
6. Else: print ONE LINE to stderr, exit 0.
```

### Output format

One line, stderr only, no color escapes (Claude Code's UI handles its own theming):

```
swanlake: project at <abs-path> has CLAUDE.md but no beacon attribution. Run 'swanlake init project --type cc' to scaffold, or 'touch <abs-path>/.swanlake-no-beacon' to silence.
```

The line is ~180 chars, intentionally long enough to carry both remediation paths so the operator never has to look up the silence command. Repeated nudges across sessions are tolerable: a hook that nags is louder than a hook that auto-fixes is dangerous.

### Exit codes

| Exit | Meaning |
|---|---|
| 0 | Always. The hook is advisory; non-zero would block the session start |

The hook MUST exit 0 even on its own internal errors (missing dependencies, malformed stdin, unreadable files). A SessionStart hook that exits non-zero blocks the session — a security-tooling hook that breaks the operator's harness fails the "don't make security expensive" test. Internal errors can write to stderr but exit 0.

### Performance budget

The hook runs at every CC session start. Hard budget: < 50ms wall-clock on a warm cache. The script is < 80 lines of bash; the only filesystem reads are `test -f` calls and one `grep` against `CLAUDE.md` (which is small). No subprocesses to swanlake itself — the nudge logic is reimplemented in bash because spawning Python at every session start is the wrong cost shape.

## State model

`swanlake scan` is read-only — it touches no state. The session-nudge hook is read-only — it touches no state. The only state-touching new code paths are:

| File | What writes | When |
|---|---|---|
| `~/.swanlake/cc-adapter-manifest.json` | `swanlake adapt cc --enable-session-nudge` adds a `session_nudge` entry; `--disable` removes it | Operator invocation only |
| `~/.swanlake/audit.jsonl` | Every `swanlake scan` and `swanlake init project` call appends one row (existing pattern from v0.2) | Operator invocation only |
| Target project tree | `swanlake init project DIR` writes `CLAUDE.md`, optional `.swanlake-no-beacon`, optional `cmas/` skeleton | Operator invocation only |

No new files under `~/.swanlake/` are introduced by v0.4. The scan results are not cached — operators expect a fresh walk every time, and the projects tree is small enough (~50 projects worst-case) that a cache would cost more in bug surface than it saves in latency.

## `.swanlake-no-beacon` opt-out semantics

v0.3 already defined the opt-out marker (`docs/swanlake-no-beacon.md`, `swanlake/commands/beacon/_optout.py`). v0.4 extends its scope from "exclude from sweep + deploy" to "exclude from scan + nudge". No format changes; the same `find_marker()` walk applies.

| Layer | Behavior on opt-out hit |
|---|---|
| L1 `swanlake scan` | Project is reported with `recommended_action: opted-out` and counted in `summary.n_optout`; not reported in `n_actionable` |
| L2 SessionStart nudge | Hook exits 0 silently. No nudge printed. |
| L3 `swanlake init project` | Refuses with `target <path> is opted out via <ancestor>/.swanlake-no-beacon; remove the marker before initialising` |

The L3 refusal is intentional: scaffolding into an opted-out tree would defeat the marker. The operator can `rm` the marker first if they actually want to convert an opted-out project into a beaconed one.

## Risk register

| # | Risk | Prob | Impact | Mitigation |
|---|---|---|---|---|
| R1 | False-positive nudges on non-agentic projects (project has `CLAUDE.md` but operator never wanted it beaconed) | High | Med | The hook respects `.swanlake-no-beacon`. The nudge string includes the silence command verbatim. Nudge is one stderr line, not a UI dialog. Operators who want zero nudges can `--disable-session-nudge` to remove the hook entirely; the alternative (auto-detecting "is this an agentic project") would require a heuristic that is itself a bug source. Drop the burden on the explicit opt-out marker. |
| R2 | Nudge spams the operator at every CC session start, becoming background noise | High | Low | Exactly one stderr line per session start. No notification, no terminal bell, no color. The string carries the silence command. This is the operator-confusion mode the marker exists to prevent (mirrors v0.3 R3 reasoning). Future iteration: a v0.4.x patch could add per-project once-per-N-days throttling via a `~/.swanlake/nudge-throttle.json` cache, but not in MVP — caches add a stale-state failure mode that exceeds the value. |
| R3 | Chicken-and-egg: the nudge hook tells the operator to run `swanlake init project`, but Swanlake itself isn't on PATH | Med | Med | The nudge string includes the literal command, not a "run swanlake". If `swanlake` isn't on PATH the operator gets a `command not found` and goes hunting — which is the right mode (Swanlake should be installed before any of v0.4 matters). The hook does not check PATH itself; that check belongs in `swanlake doctor`, which the operator runs against their actual install state. |
| R4 | Interaction with v0.3.x worktree-install structural fix (CC adapter writes via worktree-relative paths) | Low | Med | `--enable-session-nudge` lives on the same CC adapter that v0.3.x patched. The hook script path is resolved via the same `_templates_dir()` helper (`swanlake/adapters/templates/cc/hooks/`). Test suite includes a worktree-install case for the new hook. The manifest entry uses absolute paths post-install (matches existing hook behavior); no worktree-vs-real-install ambiguity at runtime. |
| R5 | `swanlake init project` writes a CC `CLAUDE.md` template that itself contains a stale Beacon Part A (canon drift) | Med | High | The template file at `swanlake/adapters/templates/init/cc/CLAUDE.md.template` does not embed Part A inline. It uses the v0.4 import directive (`@canon/operating-rules.md`) and copies `canon/operating-rules.md` from the swanlake repo into the new project's `canon/` dir as part of scaffolding. Drift is impossible because the rules file IS the canon copy. (Validation: scaffolding-time test diffs the canon file in the repo against the file in the test scaffold and refuses on drift.) |
| R6 | `swanlake scan` walks deep into vendored / submodule trees (mirrors v0.3 R7) | Med | Low | Reuse the `SKIP_DIRS` set from `swanlake/coverage.py` (`.git`, `node_modules`, `.venv`, `venv`, `__pycache__`, `.pytest_cache`, `dist`, `build`, `.mypy_cache`). Default scan is one level under `~/projects/` (project roots only); nested mode (`--include-nested`) uses rglob with the same SKIP_DIRS. |
| R7 | Operator runs `swanlake init project` against a non-empty dir and overwrites in-progress work | Med | High | Default refuses non-empty target dirs with exit 2 USAGE and a clear error: `target <path> is non-empty (N files); pass --force to overwrite, or pick an empty dir`. `--force` is the explicit confirmation; no env-var bypass. (Mirrors the v0.3 D3 decision that `--allow-dirty` was removed.) |
| R8 | The hook script is portable across CC versions, but the SessionStart event payload changes shape | Low | Med | The script reads `$CLAUDE_PROJECT_DIR` first; this is a documented harness env var (CC settings docs). Stdin JSON parsing is the fallback path — wrapped in `2>/dev/null \|\| true` so a payload schema change degrades to "use cwd" rather than "exit non-zero and block session". Test suite covers both paths plus the no-payload case. |
| R9 | `init project --type cma` creates a `cmas/` dir but Swanlake doesn't ship CMA fixture files (per N3) | Low | Low | Documented behavior. The CMA scaffold creates an empty `cmas/` dir + `zones.example.yaml` + `CLAUDE.md`. Operator populates `cmas/` themselves. The init command prints a one-line hint: `next: drop CMA definitions under cmas/ then run 'swanlake adapt cma --project <dir>'`. No "starter CMA" is bundled — that would be a curated-content surface Swanlake refuses to maintain (mirrors v0.3 N8 reasoning that registry generation stays in `make-canaries.py`, not Swanlake itself). |

## Non-goals (explicit)

The hard NO list. Each item below is a thing v0.4 will NOT do, with the reason it's excluded so a future contributor doesn't re-litigate it.

| # | Non-goal | Why |
|---|---|---|
| N1 | Auto-install / auto-scaffold on SessionStart (the L4 layer) | Same architectural rule v0.3 used to bar auto-deploy on cloud surfaces. Auto-scaffolding on detect creates a centralized attack surface on whatever triggers the auto-install. An attacker who compromises that trigger can silently scaffold attacker-known beacon content into every project. The defense's whole attribution property collapses to whatever guards the auto-install trigger. Cheapest defense: never have one. (Reinforced architecturally — there is no `nudge/auto_scaffold.py` module; adding one requires touching dispatch + this list.) |
| N2 | Auto-modify any operator surface (CLAUDE.md, settings.json, vault notes, README, gitignore) without explicit per-action operator invocation | Same as N1 generalized. The only mutative surfaces v0.4 touches are: (a) target dir of `swanlake init project DIR` (operator passed it on the command line); (b) `~/.claude/settings.json` via `swanlake adapt cc --enable-session-nudge` (operator passed the flag). Both are explicit per-action invocations. |
| N3 | CMA fixture deployment (no bundled "starter CMA" definitions) | A bundled CMA fixture tree would become a curated-content surface Swanlake has to keep current; the same reason `make-canaries.py` lives in the operator's repo, not in Swanlake. Operators who want a starter set can fork it from another project they own. |
| N4 | Cache scan results | Caches add a stale-state failure mode that exceeds the value. The walk is fast (< 200ms for ~50 projects). Future iteration could add a JSON cache under `~/.swanlake/scan-cache.json` if a real performance complaint surfaces — not in MVP. |
| N5 | Throttle nudges per-project per-session-batch | Same reasoning as N4. The simplest possible behavior (one stderr line every session start that matches conditions) is observable; the next-simplest behavior (once per N days, persisted to `~/.swanlake/nudge-throttle.json`) introduces a cache invalidation problem. The operator's escape hatch is `--disable-session-nudge` or `.swanlake-no-beacon`. |
| N6 | Walk outside `~/projects/` (or `$SWANLAKE_NUDGE_SCOPE`) | The nudge hook only fires for projects under the configured roots. Out-of-scope projects (system dirs, tmp dirs, vendored copies on `/mnt`) get silent exit 0. The default reflects the operator's documented dev-workspace location. |
| N7 | Block session start on missing beacon | The hook ALWAYS exits 0. A blocking hook on missing beacon would be a self-inflicted DoS — every operator who hadn't yet beaconed every project would lose CC sessions. Nudge means nudge. (See L2 contract.) |
| N8 | Modify the operator's existing `CLAUDE.md` files via `swanlake init project` | `init project` is for fresh projects only. Refusing on non-empty target (R7) ensures it never converts an existing project. To beacon an existing project, the operator runs `swanlake beacon deploy <surface>` — the v0.3 verb that already has the LOCAL deploy state machine and clean-tree pre-flight. |

## Decisions resolved

Twelve open decisions from the operator brief + spec-review pass were resolved before MVP build. Each line below pins the choice + the one-line rationale operators will need when re-reading the spec months from now.

- **D1: Three layers, not four** — the operator brief said "L1 / L2 / L3" with L4 explicitly listed in non-goals. Confirmed: three layers ship in MVP; L4 (auto-scaffold on SessionStart detect) is the architectural No this spec exists to enforce. (See §1, §3, N1.)
- **D2: `scan` lives at top level, not under `beacon`** — scan reports CMA-shape and opt-out state, not just beacon attribution. Putting it under `beacon` would mis-signal scope. Top-level matches the broader audit semantics. (See §4.)
- **D3: SessionStart hook is opt-in via `swanlake adapt cc --enable-session-nudge`** — Swanlake never auto-wires hooks on install. Two reasons: (a) operators who run their own production hooks must not get surprise additions; (b) consistent with v0.3 D3 (no `--allow-dirty` escape hatch — explicit invocations only).
- **D4: Hook is bash, not Python** — performance budget (§5) requires < 50ms wall-clock at every session start. Python startup blows that budget. Bash, stdlib-only (no jq dependency), keeps the budget. Logic is reimplemented from `_optout.py` semantics — the duplication is bounded (< 80 LOC) and the alternative is worse.
- **D5: Hook MUST exit 0 always** — a SessionStart hook that exits non-zero blocks the session. A security-tooling hook that breaks the operator's harness fails the "don't make security expensive" test. Internal errors write to stderr but exit 0. (Mirrors v0.3 N7's "Swanlake never auto-cleans" — be visible without being destructive.)
- **D6: `swanlake init project` requires `--type` explicitly** — no default. Forces operator to make the cc-vs-cma choice consciously. The two scaffolds differ in real ways (CMA gets `cmas/` + `zones.example.yaml`); a wrong default would be a footgun.
- **D7: `init project` refuses non-empty targets without `--force`** — strict abort, no env-var bypass. The escape hatch widens attack surface (operator may export the env var globally and never reset it). Use `--force` per-invocation. (Mirrors v0.3 D3.)
- **D8: Beacon Part A in scaffolded `CLAUDE.md` uses `@canon/operating-rules.md` import + canon-file copy, not inline embed** — drift-resistant by construction. The canon file IS the source of truth; the scaffold copies it into the new project's `canon/` dir. (R5 mitigation.)
- **D9: Scan default scope is one level under `~/projects/` (project roots only)** — nested layouts (monorepos) opt in via `--include-nested`. Reason: the common case is "which of my top-level projects need attention"; the deep walk is opt-in to keep the default fast and noise-free.
- **D10: No L2 throttling cache in MVP** — see N5. Simplest behavior is observable; throttling adds invalidation bugs. Operator escape hatch is the opt-out marker or `--disable-session-nudge`.
- **D11: `$SWANLAKE_NUDGE_SCOPE` env var overrides the default `~/projects/` scope** — operators with non-standard layouts (e.g. `~/code/`, `~/dev/`) can repoint without editing the hook. Defaults to `$HOME/projects` if unset.
- **D12: Audit log appends one row per `swanlake scan` and `swanlake init project` invocation** — same schema as v0.2. The hook does NOT append to the audit log (the hook's whole point is "be cheap"; opening fcntl-locked log files at every session start would blow the perf budget and add a failure mode).
