# Swanlake v0.3.x — worktree install-isolation spec

> Status: design-locked, MVP shipping in this PR. Single source of truth for the v0.3.x install-isolation patch. Ships independently of the v0.3 beacon-deploy work.

## Headline

Through v0.3.0 the Swanlake CLI is installed editable (`pip install -e .`) against whichever source tree the operator (or any background agent) last ran the install from. Editable installs write a single per-environment `.pth` finder that points at one absolute path. That path is **process-global to the user**: every shell on the machine, every cron job, every CI invocation, and every other Claude Code agent in any other worktree all import `swanlake` from the same place.

When a background build agent runs `pip install -e .` inside its own worktree at `~/projects/Swanlake/.claude/worktrees/<branch>/`, it silently captures the global editable pointer. The next `swanlake --version` invocation — from any shell, including the operator's stable one — reports whatever `__version__` that worktree's source declares. The bug surfaces as "the CLI lies about its version", but the underlying class is broader: **any agent's worktree edits become live for the operator without the operator opting in.**

This is not a setuptools bug. It is the documented behavior of editable installs. The fix is to surface the drift loudly, and to recommend an installation discipline where agent worktrees never share an interpreter with the operator's stable install.

## Reproduction

```bash
# Operator's stable shell, main checkout, v0.3.0 installed editable.
$ swanlake --version
swanlake 0.3.0

# Background agent in a worktree:
$ cd ~/projects/Swanlake/.claude/worktrees/v0.4-mvp/
$ # Agent edits swanlake/__init__.py: __version__ = "0.4.0-dev"
$ pip install -e . --break-system-packages
Successfully installed swanlake-cli-0.4.0-dev

# Operator's stable shell, no action taken:
$ swanlake --version
swanlake 0.4.0-dev          # silently reports the agent's in-progress version
```

The pointer drift is visible in `__editable___swanlake_cli_*_finder.py` under
the user's `site-packages`: the `MAPPING` dict carries one absolute path per
package and gets overwritten on every `pip install -e .`.

## Problem statement

| Property | Current behavior | Required behavior |
|---|---|---|
| Source of truth for the installed CLI | Whichever worktree most recently ran `pip install -e .` | The worktree the operator deliberately installed from |
| Operator visibility into drift | None — `swanlake --version` reports whatever `__init__.py` the finder lands on | A loud, single-line stderr warning the first time the CLI runs from a different source than the install marker |
| Agent invariant inside its worktree | Agent's `pip install -e .` silently mutates the operator's CLI | Agent's `pip install -e .` either (a) is opt-in scoped via venv/pipx, or (b) leaves a marker the next operator-shell invocation flags |
| Doctor probe coverage | None for install drift | One probe row that fails when the runtime path does not match the marker |

The class of failure is "agent runtime mutates operator runtime through a shared editable install." The defense is to make the mutation non-silent, and to document a stronger isolation model for operators who want zero risk.

## Solutions surveyed

Six approaches considered. The recommendation at the end is a layered combo of two of them.

| # | Approach | How | Pros | Cons | Cost |
|---|---|---|---|---|---|
| 1 | Per-worktree venv | Each worktree carries its own `.venv/`; agents activate it before any `pip install` | Hard isolation; bug becomes impossible by construction | Requires every agent + every operator shell to remember the activation step. Automation that fails when humans forget is theater. Adds 50-200 MB per worktree | M (one-time docs, ongoing discipline) |
| 2 | Install from PyPI tarball, not editable | `pip install swanlake-cli==0.3.0` instead of `-e .` | Frozen at the installed version; immune to source mutations | Loses the "edit-and-test-immediately" loop for the operator's own dev work. Forces a release cycle for every change the operator wants to use | M for operator workflow |
| 3 | Install-marker file check | On `pip install -e .` write `~/.swanlake/.install-marker` with the source path. CLI startup compares the running module's source directory to the marker; warns on mismatch | Stdlib-only, no venv discipline required, works for the existing editable install model | Warning only — does not prevent the drift, just surfaces it | S (one PR, fits in v0.3.x) |
| 4 | pipx-based install | Operator installs via `pipx install swanlake-cli` — pipx isolates each tool in its own venv | Hard isolation like (1) but only for the tool itself, not for every agent worktree. Standard pattern for CLI tools | Requires pipx on the system. Loses editable-install workflow for operator dev work; operator who edits source must `pipx install -e <path>` deliberately | S for end-users / M for operator dev workflow |
| 5 | `__version__` from git, not file | Drop `__version__` from `__init__.py`; compute at runtime via `git describe --tags` | Always correct vs. the source tree; no marker file needed | Fixes the version-reporting symptom but not the broader class (agent edits to any other module still go live silently). Adds a git subprocess to every CLI invocation. Breaks in non-git installs (PyPI tarball) | M, partial fix |
| 6 | Worktree-scoped install via PYTHONPATH | Agent prepends its worktree to `PYTHONPATH` instead of running `pip install -e .` | No persistent state mutation — scoped to the agent's process tree | Requires every agent to know about and use this pattern. Does not solve the bug that already happened (a shared install already exists); only prevents future cases | M (requires agent-runtime cooperation) |

### Why not just (1) or (4)

Both fix the bug by construction. Both require humans (or every Claude Code subagent harness) to remember the activation step. The user's standing rule on this is explicit: **"automation that fails when humans forget is theater."** Per-worktree venvs and pipx are excellent recommendations for the operator's deliberate workflow, but the v0.3.x patch must do something useful for the operator who has not yet adopted them — and for every operator who clones the repo and runs `pip install -e .` once without thinking about isolation.

### Why not just (2) or (5)

(2) breaks the operator's edit-loop. (5) only fixes the version-string symptom; agent edits to any other file (e.g. a probe in `doctor.py`, an exit-code constant, a hook script) still go live silently for the operator and report the same `__version__`.

## Recommendation — layered (3) + (4)

Ship **(3) install-marker check** as the in-CLI MVP fix, and **document (4) pipx as the recommended install path** for operators who want hard isolation. Both can land together; neither blocks the other.

**(3) is the load-bearing defense for the existing user base** — the operator and every CI consumer who already `pip install -e .`'d Swanlake gets a loud warning the first time their CLI runs from a different source than the marker. No workflow change required.

**(4) is the prevention story for future installs** — `pipx install swanlake-cli` puts the tool in its own venv, so any agent's `pip install -e .` inside a worktree affects only that agent's interpreter, not the operator's `swanlake` binary. The README install section recommends pipx as the default and mentions `pip install -e .` as the dev-loop alternative.

(1), (2), (5), (6) remain documented in the spec as "considered, rejected for these reasons" so the design conversation does not get re-litigated next quarter.

## Implementation punch list

### T1. Install-marker write (build-time)

On `pip install -e .` (and `pip install .`), setuptools must drop `~/.swanlake/.install-marker` with the absolute path of the source directory it was invoked from. Stdlib-only; no new dependencies.

Mechanism: a `[tool.setuptools.cmdclass]` entry that subclasses `develop` and `install` to invoke a single helper. The helper:

- Resolves `~/.swanlake/` (creating it mode 0700 if absent — same logic as `swanlake.state.ensure_state_root`)
- Writes `.install-marker` mode 0600 with two fields, one per line:
  - `source_path=<absolute path of the source dir setup.py was invoked from>`
  - `installed_at=<UTC ISO-8601 timestamp>`
- Never raises into setuptools — write failures degrade to a stderr warning. The install must not fail because the marker write failed (e.g. read-only home directory in a container).

The marker file is an undocumented internal contract between the build hook and the CLI; the format is two `key=value` lines, no TOML/JSON ceremony.

**Edge case (release tarball install):** when installed from PyPI tarball or `pip install .` from a checkout (non-editable), the source path is a temporary build directory that pip will delete after install. In that case the marker writes the *checkout* path if discoverable from `os.getcwd()` matching the unpacked sdist root, otherwise the marker is omitted entirely. The CLI startup check treats a missing marker as "no drift to detect" (silent), not as an error.

### T2. CLI startup mismatch warning

In `swanlake/cli.py`, before `_dispatch`, add a one-shot check:

- Resolve the running module's source directory via `Path(swanlake.__file__).resolve().parent.parent`
- Read `~/.swanlake/.install-marker`; parse the two-line key=value format
- If the marker exists and its `source_path` does not match the running module's source directory, print a single stderr line:

```
warning: swanlake CLI is running from <runtime-path>
         but the install marker at ~/.swanlake/.install-marker points at <marker-path>.
         This usually means a background agent ran `pip install -e .` inside its own worktree.
         Run `pip install --force-reinstall <intended-source>` to reset, or use pipx for isolation.
```

The warning prints once per process. Suppressed under `--quiet`. Suppressed under `SWANLAKE_NO_INSTALL_DRIFT_WARN=1` for CI use cases where the divergence is expected (e.g. testing the CLI from a feature-branch worktree). Always suppressed when reading the marker raises any exception — the warning itself must never crash the CLI.

### T3. Doctor probe — install marker

Add a 10th probe to `swanlake/commands/doctor.py`:

| Probe | pass | warn | fail |
|---|---|---|---|
| `install-marker matches runtime` | marker exists and matches `Path(swanlake.__file__)` | marker missing (e.g. installed via pipx, or installed before this version shipped) | marker exists and does not match — likely worktree-install drift |

The probe shows the marker source path, the runtime source path, and the diff in the `--fix-suggestions` output:

```
fix: pip install --force-reinstall <intended-source>
     (or migrate to pipx: pipx install swanlake-cli)
```

### T4. README documentation

Add an "Install" section to `README.md` recommending pipx as the default install path:

```bash
pipx install swanlake-cli                              # recommended; isolated venv
pip install --user swanlake-cli                        # alternative; shared user site-packages
pip install -e .                                       # dev loop only; see worktree-isolation note
```

Cross-reference `docs/v0.3.x-worktree-install-isolation-spec.md` from the dev-loop bullet. Two-paragraph note warning operators to use a per-worktree venv (or pipx) when running multiple background agents in parallel worktrees.

### T5. Tests

| Test file | Cases |
|---|---|
| `swanlake/tests/test_install_marker.py` | (a) marker write helper creates `~/.swanlake/.install-marker` mode 0600 with both fields. (b) marker write degrades to silent stderr on unwritable state root. (c) marker parser tolerates trailing whitespace, comments, missing fields without raising |
| `swanlake/tests/test_cli_install_drift_warning.py` | (a) no marker → no warning, exit 0. (b) marker matches runtime → no warning, exit 0. (c) marker mismatches runtime → warning printed once to stderr, exit code unchanged. (d) `--quiet` suppresses the warning. (e) `SWANLAKE_NO_INSTALL_DRIFT_WARN=1` suppresses the warning. (f) marker read raises → no warning, no crash |
| `swanlake/tests/test_doctor_install_marker_probe.py` | (a) probe pass when marker matches. (b) probe warn when marker missing. (c) probe fail when marker mismatches. (d) probe surfaces both paths in `--fix-suggestions` output |

All three test files use `tempfile.TemporaryDirectory()` + `_state.set_state_root()` — same pattern as `test_doctor_cmd.py`. No mocking of file paths; the state-root override is the seam.

### T6. Audit log field

Augment the audit row schema (`swanlake/audit.py`) with one new field: `install_marker_source` set to the parsed `source_path` from the marker (or `null` if missing/unreadable). Drift events are now post-hoc reproducible from the audit log alone — the operator can grep for rows where `install_marker_source != <expected>` to find every command that ran under the wrong install.

(T6 is in scope for this PR if the audit module is small enough to touch safely; otherwise it slips to v0.3.x.1 and the spec marks it as deferred.)

## Risk register

| ID | Risk | Likelihood | Mitigation |
|---|---|---|---|
| R1 | Setuptools cmdclass hook breaks `pip install -e .` for the existing user base | Low — the helper has try/except around the entire write; install never fails on marker write failure | T5(b) test asserts the helper degrades cleanly. CI runs `pip install -e .` against a tmpdir HOME to verify |
| R2 | Marker write to `~/.swanlake/.install-marker` collides with a real swanlake state file | None — `.install-marker` is a new filename, dotfile prefix to keep it out of `ls`-default output | n/a |
| R3 | Multiple Python interpreters on the same machine (3.11 vs 3.12) get separate site-packages but share `~/.swanlake/.install-marker`, causing false-positive warnings | Medium — operator may have both Python versions and intentionally install Swanlake in both | Marker stores `python_executable=<sys.executable>` as a third field; the CLI compares marker's `source_path` only when the current `sys.executable` matches the marker's. Cross-interpreter installs do not trigger the warning |
| R4 | The check fires during `swanlake --version` (which is `argparse` SystemExit before reaching `_dispatch`), defeating the purpose | High if naive | Place the marker-check call **before** `parser.parse_args()` runs, so it executes even when the argv triggers an action='version' early-exit. Test (T5 c) asserts `swanlake --version` from a drifted install still warns |
| R5 | Operator legitimately installs Swanlake from two separate clones (e.g. v0.3.x maintenance branch + main dev) and gets warned every shell | Low — uncommon workflow | `SWANLAKE_NO_INSTALL_DRIFT_WARN=1` env var is the documented escape hatch. Doctor probe still flags the drift, so the operator stays informed when they `swanlake doctor` |
| R6 | Background agent's `pip install -e .` in a worktree clobbers an operator's pipx install of swanlake-cli | None — pipx installs to its own venv with its own site-packages; agents running `pip install -e . --user` cannot reach the pipx venv | n/a — this is exactly why pipx is the recommended path |
| R7 | The marker file becomes a tempting target for an attacker (e.g. write a malicious `source_path` to redirect the warning) | Negligible — the marker only feeds an informational stderr warning. No code is loaded from `source_path`; the warning text is the only consumer. The marker is mode 0600 in a 0700 state dir | Threat-model row added under THREAT-MODEL.md "Install integrity" |

## Non-goals

The following are **explicitly not addressed** in this spec. Each has a rationale; none are open questions for v0.3.x.

- **Hard prevention of editable-install drift.** The marker is a warning, not a lock. Operators who want hard prevention adopt pipx (recommended) or per-worktree venvs (also documented). Swanlake does not patch pip.
- **Auto-rewriting the editable finder's `MAPPING` dict.** Mutating site-packages from inside the CLI is exactly the kind of action that breaks for the next operator and would itself need to be reverted. The marker check observes; it does not intervene.
- **Cross-machine drift detection.** The marker is per-host. Operators running Swanlake across multiple hosts (laptop + server) get one marker per host and one warning per host. No central registry.
- **Telemetry of drift events.** No phone-home, no usage analytics. Drift is reported only via local stderr + `swanlake doctor` + the audit log. Consistent with N5 (no phone-home).
- **Solving the broader "shared editable install" pattern for other Python tools.** This spec is scoped to Swanlake. The pattern (loud warning + pipx recommendation) generalizes; documenting that generalization is out of scope.
- **Backporting to v0.2.x.** The marker write requires a setuptools cmdclass change in `pyproject.toml`, which v0.2.x maintenance does not touch. v0.2.x users who want the warning upgrade to v0.3.x.

## Migration

The marker is created by the next `pip install -e .` (or `pip install .`) after this PR ships. Operators who already have v0.3.0 installed and never re-install get no marker — and therefore no warning — until they next reinstall. That is the intended degrade-to-silent behavior: the warning never fires for an operator who has not opted into v0.3.x.

`swanlake doctor` will show the install-marker probe as `warn (marker missing)` for those operators, with the suggested fix `pip install --force-reinstall <path>` (or `pipx install swanlake-cli`) to populate it.

## Out of scope, parked for v0.4

- `swanlake doctor --repair install-marker` to write the marker post-hoc without a full reinstall. Useful for the "I have v0.3.x but the marker is missing" case. Cheap to add; deferred to keep this PR small.
- Replacing the editable install entirely with a `swanlake-dev` shim that resolves the source path at runtime via an env var. Larger change, broader blast radius, not justified by the bug class.
